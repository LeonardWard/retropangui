# Odroid C5 레트로 게임 OS 프로젝트 분석 보고서 (2026-05-05)

본 문서는 Odroid C5 기반 레트로 게임 OS 프로젝트의 현재 구성을 검토하고 발견된 주요 문제점과 개선 방향을 정리한 보고서입니다.

---

## 1. 그래픽 스택 의존성 및 링킹 문제
현재 시스템은 Mali-G310 전용 바이너리 드라이버(`mali-ddk`)와 오픈소스 그래픽 스택(`mesa3d`)이 혼재되어 있습니다.

*   **현상:** `retroarch`, `emulationstation`이 `mesa3d`를 직접 의존성으로 참조하고 있음.
*   **문제점:** 
    *   Buildroot의 가상 패키지(Virtual Package) 메커니즘(`libegl`, `libgles`)을 사용하지 않아 빌드 타임 의존성이 Mesa에 고정됨.
    *   Mali 래퍼가 런타임에 심볼을 대체하지만, 빌드 타임의 헤더 불일치로 인해 특정 API 호출 시 크래시 가능성이 큼.
*   **개선 방향:** `mali-ddk`가 `libegl`, `libgles`, `libgbm`의 제공자로 설정되도록 수정하고, 에뮬레이터 패키지들의 의존성을 추상화된 가상 패키지로 변경.

## 2. 빌드 환경 종속성 (Hardcoded Paths)
빌드 스크립트 내에 특정 개발 환경에만 존재하는 절대 경로가 포함되어 있습니다.

*   **현상:** `board/odroidc5/post-build.sh` 내에 `/home/builder/themes` 경로 존재.
*   **문제점:** 다른 사용자의 환경이나 CI/CD 시스템에서 빌드 시 테마 복사 단계에서 실패함.
*   **개선 방향:** `${BR2_EXTERNAL_C5_PANGUI_PATH}` 또는 상대 경로를 사용하여 프로젝트 내부 자원을 참조하도록 수정.

## 3. 파티션 구조 및 데이터 보존성
현재의 파티션 레이아웃은 데이터 관리와 업데이트 측면에서 불리합니다.

*   **현상:** `BOOT`와 `ROOTFS` 단일 파티션 구조. ROM 경로가 RootFS 내부에 위치.
*   **문제점:** 
    *   OS 이미지 업데이트 시 유저의 ROM 데이터가 삭제됨.
    *   RootFS 크기가 고정되어 있어 용량 확장이 불편함.
*   **개선 방향:** 별도의 `SHARE` 파티션(exFAT)을 정의하고, 첫 부팅 시 남은 SD 카드 용량을 자동으로 확장 및 마운트하는 로직 추가.

## 4. 네트워크 서비스 중복
불필요한 중복 서비스가 활성화되어 있습니다.

*   **현상:** `dropbear`와 `openssh`가 동시 활성화.
*   **문제점:** SSH 포트(22번) 점유 경쟁 및 부팅 로그 지저분해짐.
*   **개선 방향:** 시스템 경량화를 위해 `dropbear` 하나로 통합 권장.

## 5. U-Boot 바이너리 관리 및 재현성
바이너리 형태의 U-Boot 사용으로 인해 빌드 과정이 불투명합니다.

*   **현상:** `board/odroidc5/u-boot.bin.sd.bin` 파일을 직접 사용.
*   **문제점:** Buildroot가 소스에서 빌드한 U-Boot가 반영되지 않으며, Amlogic 특유의 서명(FIP) 과정이 생략되어 있어 유지보수가 어려움.
*   **개선 방향:** `post-image.sh`에서 하드커널 제공 툴을 사용하여 빌드된 `u-boot.bin`을 자동으로 서명하고 이미지를 생성하도록 자동화.

## 6. 실행 제어 및 권한 설정
시스템 부팅 후 사용자 경험과 보안 설정이 미비합니다.

*   **현상:** 
    *   EmulationStation 자동 실행 스크립트(init.d) 미확인.
    *   모든 프로세스가 `root` 권한으로 실행될 위험.
*   **개선 방향:** 
    *   `/etc/init.d/S99emulationstation` 추가.
    *   전용 유저 생성 및 권한 그룹(video, input, audio) 할당.
